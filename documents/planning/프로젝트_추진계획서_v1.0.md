## 1. 사업(프로젝트) 개요

### 1.1 배경/목적
- **배경**: 지역 상권 활성화 및 사용자 걷기 참여를 연결하는 리워드 구조 필요
- **목적**: “걷기 → 미션 달성 → 쿠폰 획득/사용” 흐름을 MVP로 구현하고 출시까지 완료

### 1.2 목표(Outcome)
- 사용자 관점 핵심 플로우 완성(인증 → 미션/쿠폰 → 지도/매장 → 푸시)
- 운영 관점 최소 운영 요건 확보(정책/로그/권한/배포 절차)

### 1.3 성공 지표(예시)
- 가입 전환율, 미션 완료율, 쿠폰 사용 전환율, D7/D30 재방문율

---

## 2. 범위 정의

### 2.1 MVP 1차(출시 범위)
- **인증**: 이메일/비밀번호 + 카카오 로그인(Custom Token)
- **쿠폰**: 쿠폰 목록/상세/사용(4자리 코드), 유효기간/중복사용 방지
- **미션**: 걸음 수 기반 미션(진행률/완료/보상으로 쿠폰 지급)
- **지도/매장**: 주변 매장 표시/검색(최소 기능), 매장 상세
- **알림**: 쿠폰/미션/공지 푸시(FCM)

### 2.2 MVP 2차(출시 제외/후속)
- 친구 초대/룰렛/구독/결제(Bootpay) 등은 **일정/리스크**에 따라 2차로 분리 가능

---

## 3. 추진 전략(원칙)

### 3.1 개발 방식
- **주 단위 마일스톤**으로 기능 완결(주간 데모/검토 → 다음 주 반영)
- **검증 로직은 서버(Functions) 중심**(쿠폰 사용/발급, 미션 완료 검증 등 치팅 방지)

### 3.2 기술 원칙(요약)
- Flutter + Riverpod + GoRouter 기반 표준 구조 유지
- Firestore 규칙은 **최소 권한** 원칙, 민감 처리 Functions로 이동
- 로그/이벤트는 운영/분석에 활용 가능하도록 최소 스키마 확보

### 5.2 주차별 핵심 범위(요약)
- **W1**: 킥오프, 요구사항 확정, 일정/리스크 합의, 인증/기반 세팅
- **W2**: 앱 정보구조/화면흐름/데이터 규칙 설계 + 인증 플로우 구현
- **W3**: 쿠폰(화면+서버 검증) 1차 완결
- **W4**: 미션(걸음 수 연동/진행률/보상) 1차 완결
- **W5**: 지도/매장/푸시 + 통합/회귀 테스트 시작
- **W6**: QA/버그 수정/배포/검수/오픈 + 운영 인수인계

### 5.3 마일스톤(예시)
- M1: 요구사항/범위 확정(분석검토)
- M2: 설계 산출물 확정(설계검토)
- M3: 핵심 기능 데모(쿠폰/미션)
- M4: 기능 동결(Feature Freeze)
- M5: 스토어 제출/심사 대응
- M6: 출시/검수 완료

## 7. 품질/검수 계획

### 7.1 완료 기준(Definition of Done)
- 기능별 **시나리오 기반 체크리스트**를 통과
- 서버 검증이 필요한 기능은 Functions 기준으로 **치팅/중복/권한** 처리 완료
- 크래시/치명 버그 0, 주요 플로우 이탈 0을 목표
- 릴리즈 빌드(테스트 트랙/TestFlight) 배포 가능 상태

### 7.2 테스트 범위(권장)
- 단위/위젯 테스트(핵심 로직/포맷/검증)
- 통합 테스트(인증→미션→쿠폰→사용)
- 기기 테스트(저사양 Android, iOS 1대 이상)

### 8.1 주요 리스크(예시)

- 변경 요청은 “요구사항/일정/리스크” 영향도를 1페이지로 정리 후 승인
